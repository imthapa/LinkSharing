What?

Problems faced while designing a software

  • Designing object-oriented software is hard
  • Designing reusable object-oriented soft-
ware is even harder
  • Documenting a design is the hardest

What is a Design Pattern?

  • Well-proved solution for solving the specific problem/task
  • Programming language independent strategies for solving the common object-oriented design problems
  • Represents an idea, not a particular implementation
  • Descriptions of communicating objects and classes that are customised to solve a general design problem in a particular context.

Four essential elements of a design patterns

  • Intent
  • Solution

Why Design Patterns?

Because Someone has already solved your problem!!

Why Design Patterns?

  • Provide you with a way to solve issues related to software development using a proven solution.
  • Make communication between designers / developers more efficient.
  • Are language neutral and so can be applied to any language that supports object-orientation.
  • Capture expert knowledge and design trade-offs and make expertise widely available.

Strategy Pattern

Intent

  • Have multiple algorithm for a specific task and client decides the actual implementation to be used at runtime.

  • Lets the algorithms vary independently from clients that use it.
  • Design is open for future extension.

Structure Diagram

Participants

  • Strategy
​Declares an interface common to all supported algorithms. Context uses this interface to call the algorithm defined by a ConcreteStrategy.
  • Concrete Strategy
Implements the algorithm using the Strategy interface.

  • Context
maintains a reference to a Strategy object to make a call to the algorithm

Collaborations

  • Strategy and Context interact to implement the chosen algorithm.
  • Clients usually create and pass a ConcreteStrategy object to the context; thereafter, clients interact with the context exclusively.

  • There is often a family of ConcreteStrategy classes for a client to choose from.

Use When

  • The only difference between many related classes is their behavior.
  • Multiple versions or variations of an algorithm are required.
  • Algorithms access or utilize data that calling code shouldn't be exposed to.
  • The behavior of a class should be defined at runtime.

Considerations

  • Families of related algorithms
  • An alternative to subclassing
  • Eliminate conditional statements
  • Provide different implementations of the same behavior
  • Clients must be aware of different Strategies
  • Communication overhead between Strategy and Context
  • Increased number of objects

Observer Pattern

Intent

  • Defines a one-to-many relationship between a set of objects, where many objects are dependent on one object.

  • When state of one object changes, all of its dependants are notified and/or updated automatically.

Structure Diagram

Participants

  • Subject
​- Knows its observers.
- Any number of Observer objects may observe a subject.
- Provides an interface for attaching and detaching Observer
  objects.
  • Observer
- Defines an updating interface for objects that should be
  notified of changes
  in a subject.

Participants

(Cont..)

  • ConcreteSubject
​- Stores state of interest to ConcreteObserver objects.
  • ConcreteObserver
- Maintains a reference to a ConcreteSubject object.
- Stores state that should stay consistent with the 
  subject's.
- Implements the Observer updating interface to keep its
  state consistent with the subject's.

 

Collaborations

  • ConcreteSubject notifies its observers whenever a change occurs.
  • A ConcreteObserver object may query the subject for information.
  • If the relationship between subject(s) and observers becomes complex, then it is better to introduce a Change-Manager which takes care of notify and update operations.

Use When

  • State changes in one or more objects should trigger behavior in other objects.
  • Broadcasting capabilities are required.
  • There is need to add observers without modifying the subject or other observers.
  • Loose coupling is required, so that related classes can be reused independently of each other.

Considerations

  • Unexpected updates
  • Mapping subjects to their observers
  • Observing more than one subject
  • Who triggers the update?
  • Dangling references to deleted subjects
  • The push and pull models
  • Encapsulating complex update semantics

Decorator Pattern

Intent

  • Attach additional responsibilities to an object dynamically.

  • Decorators provide a flexible alternative to subclassing for extending functionality.

Structure Diagram

Participants

  • Component
​- Defines the interface for objects that can have responsibilities added
  to them dynamically
  • ConcreteComponent
- Defines an object to which additional responsibilities can be attached
  • Decorator
- Maintains a reference to a Component object
- Defines an interface that conforms to Component's interface.
  • ConcreteDecorator
- Adds responsibilities to the component.

Collaborations

  • Decorator forwards requests to its Component object.
  • The decorator adds its own behaviour either before and/or after delegating to the object it decorates to do the rest of the job.
  • A decorator can itself be treated as a component and wrapped by another decorator.

Use When

  • Object responsibilities and behaviours should be dynamically added, without affecting other objects.
  • Responsibilities can be withdrawn
  • Subclassing to achieve modification is impractical or impossible.

Considerations

  • Provides more flexibility than static inheritance.
  • Specific functionality does not reside high in the object hierarchy. 
  • Keep Component classes lightweight.
  • If you have code that relies on the concrete component’s type, decorators will break that code.
  • Lots of little objects.

Singleton Pattern

Intent

  • Ensure a class only has one instance, and provide a global point of access to it.

Structure Diagram

Participants

  • Singleton
​- Defines an Instance operation that lets clients access
  its unique instance.
- Is responsible for creating its own unique instance

Collaborations

  • Clients access a Singleton instance solely through Singleton's Instance operation.

Use When

  • Exactly one instance of a class is required.
  • Single instance is accessible to clients from a global access.

Considerations

  • Controlled access to sole instance.
  • Permits refinement of operations and representation.
  • Permits a variable number of instances.
  • More flexible than class operations.
  • Ensuring single instance in multi-threaded context.

How?

How to select a Design Pattern?

  • Consider which desing problem does the patterns solve
  • Relate your problem with design pattern's intent
  • Identify relationships between design patterns
  • Go through multiple patterns of like purpose
  • Examine the cause of redesign
  • Consider what should be variable in your design

How to use a Design Pattern?

  • Read the pattern once through for an overview
  • Study the Structure, Participants, and Collaborations
  • Look at the Sample Code to see a concrete example of the pattern
  • Choose names for pattern participants that are meaningful in the application context

How to use a Design Pattern? (Cont..)

  • Define the classes and their relationships
  • Define application-specific names for operations in the pattern
  • Implement the operations to carry out the responsibilities and collaborations
  • Design patterns should not be applied blindly
